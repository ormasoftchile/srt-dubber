## Objective

Allow the user to select a specific audio input device by index so the correct capture device is used when the system default is not the desired microphone.

## Problem

Mac mini (and other systems without a built-in microphone) may have multiple capture devices enumerated. Without a selection mechanism the app always opens the system default, which may be AirPods Max in Bluetooth HFP/SCO mode rather than the preferred ZoomAudioDevice or USB mic.

## Changes Made

### `src/audio/recorder.hpp`
- Constructor changed to `explicit AudioRecorder(int device_index = -1)`
- Added `int m_device_index` private member field

### `src/audio/recorder.cpp`
- In `start()`: when `m_device_index >= 0`, opens a temporary `ma_context`, enumerates capture devices via `ma_context_get_devices()`, copies `ma_device_id` at that index, passes `&selected_id` to `dev_cfg.capture.pDeviceID` before `ma_device_init()`; uninits context immediately after
- Falls back to default device with a stderr warning when index is out of range
- When `m_device_index == -1`: `pDeviceID` remains `nullptr` — identical to previous behaviour

### `src/tui/app.hpp`
- Constructor gains `int device_index = -1` parameter
- Stores value to pass to `AudioRecorder`

### `src/tui/app.cpp`
- Passes `device_index` directly to `AudioRecorder` in member initialiser list

### `main.cpp`
- Parses `--device N` (two tokens) before positional SRT/video args using a `first_pos` offset
- Prints `[audio] Use --list-devices to see available input devices` hint to stderr before TUI launch
- Updated usage message to document `--device N`

## Outcomes

- ✅ User can select input device by index: `srt-dubber --device 1 input.srt`
- ✅ Default behaviour (no flag) unchanged — `pDeviceID = nullptr`
- ✅ Out-of-range index falls back gracefully with stderr warning
- ✅ Warm-up and Bluetooth fix logic untouched
- ✅ `recording_screen.cpp` required no changes — encapsulation held
- ✅ Build clean (no regressions)
